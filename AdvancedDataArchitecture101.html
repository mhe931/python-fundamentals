L1: Data Architecture Concepts (DA)

DA is the **Blueprint and Governance** of an organization's data assets.

  • It defines *what* data is collected and *how* it flows.
  • It ensures **Consistency, Quality, and Availability**.
Fun Fact: A good DA strategy can reduce data integration costs by up to 30%.

DA Roles: Comparing Responsibilities

Role Primary Focus Key Output
**DA (Architect)** Blueprint, Governance, Strategy Models (Conceptual/Logical)
**DE (Engineer)** Infrastructure, Pipelines, Data Flow ETL/ELT Scripts, Databases
**DS (Scientist)** Consumption, Insights, Algorithms Machine Learning Models, Reports

Models and Guiding Philosophy

Data Models (The Progression)

Conceptual $\rightarrow$ Logical $\rightarrow$ Physical

Entity $\rightarrow$ Attribute/Key $\rightarrow$ Data Type/Path

Guiding Philosophy: Form Follows Function

**Principle:** The design (Form) must be driven by the business need (Function).

  • **Example:** If the function is **real-time fraud detection**, the form must be a **streaming architecture** (e.g., Kafka).
  • A batch architecture would not be suitable for this function.

L2: Foundational Principles

Core principles ensuring robust data architecture:

  • **Trust:** Accuracy and reliability of data.
  • **Scale:** Ability to handle growth in volume and velocity.
  • **Modularity:** Decoupled systems for flexibility and resilience.
Key Insight: Data Architecture is less about the tools and more about the principles used to organize data.

Trust and Data Ownership

Single Source of Truth (SSOT)

  • **Definition:** For every data element (e.g., Customer Name), there is **one master system** designated as the definitive, trustworthy origin.
  • **Goal:** Eliminates discrepancies and conflicting reports across departments.

Data as a Product

Data should be managed, owned, and measured for utility just like a software product.

  • Requires clear **ownership**, established **SLAs (Service Level Agreements)**, and continuous **quality checks**.

Scale, Loose Coupling, and Frameworks

Scalability via Loose Coupling

  • **Definition:** Systems are designed to be independent of each other.
  • **Mechanism:** Often achieved using a **Broker Pattern** (Pub-Sub), where systems communicate indirectly through a message broker (e.g., a queue).
  • **Benefit:** Prevents core systems from being overloaded by downstream consumers.

L3: Data Warehousing (DWH)

DWH is the centralized repository for **structured data** built for **OLAP** (Online Analytical Processing).

  • Optimized for **read performance** and complex, historical querying.
  • Focuses on **BI (Business Intelligence)** and reporting.
Fun Fact: The concept of the Data Warehouse was formally defined in the late 1980s by Bill Inmon and Ralph Kimball.

DWH vs. Data Lake (Classic View)

Feature DWH Data Lake
**Data Type** Structured Only All Types (Raw, Unstructured, Structured)
**Process** ETL (Extract, Transform, Load) ELT (Extract, Load, Transform)
**Schema** Schema-on-Write (**Rigid**) Schema-on-Read (**Flexible**)

Note on ETL

The **Staging Area** is where the "**T**" (Transformation) primarily happens, cleaning and conforming data before the final load into the DWH.

Model and OLAP Operations

The Star Schema Model

The standard model for DWH:

**Fact Table** (Metrics/Measures) ‚Üî **Dimension Tables** (Context/Who, What, When, Where)

OLAP Operations (Multidimensional Analysis)

  • **Roll-Up ($\uparrow$):** Aggregating data to **less detail** (e.g., daily sales $\rightarrow$ monthly sales).
  • **Drill-Down ($\downarrow$):** Navigating to **more detail** (e.g., total sales $\rightarrow$ sales by individual item).
  • **Slice/Dice:** Filtering data across one (Slice) or multiple (Dice) dimensions.

L4: Data Lakes and Modern DW

A Data Lake is a centralized repository that stores all data at scale, regardless of its structure.

  • It is the home for **variety** and **raw, unexplored** data.
  • It enables **Data Science** and advanced analytics use cases.
Fun Fact: Data Lakes were initially built on low-cost object storage like Hadoop HDFS or AWS S3.

Design Approach and Key Risk

Design: The Bottom-Up Approach

The Lake processes data in stages:

Ingest $\rightarrow$ Analyze $\rightarrow$ Structure (only *after* analysis determines the need)

The Data Swamp Risk ‚ö†Ô∏è

**Definition:** A Data Lake that becomes unusable due to a total lack of organization.

**Cause:** Insufficient **Metadata** (data about the data) and **Governance** (rules on usage/quality).

Without metadata, data cannot be found or trusted.

Modern Solutions

Solution 1: Layered Architecture

Applying structure through stages of refinement:

  • **Raw (Bronze):** Unmodified, time-stamped source data.
  • **Cleaned (Silver):** Standardized, quality-checked, and integrated data.
  • **Curated (Gold):** Aggregated, modeled, and ready for BI/reporting (DWH-like).

Solution 2: The Data Lakehouse

**Convergence:** Blends the best of DWH and Data Lake.

**Key Feature:** Adds DWH features (**ACID transactions** and **Schema Enforcement**) directly to the Data Lake's low-cost storage.
